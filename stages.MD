## Этапы разработки FinFlow

### Этап 1. Подготовка проекта

**Цель:** создать основу проекта.

Что сделать:

* создать GitHub-репозиторий;
* выбрать структуру: multi-module или отдельные сервисы;
* настроить Java 21;
* настроить Gradle/Maven;
* добавить общий `docker-compose.yml`;
* поднять PostgreSQL, Kafka, Redis, ClickHouse;
* создать базовый README.

Результат:

```text
Проект запускается локально, инфраструктура готова.
```

---

### Этап 2. Auth Service

**Цель:** реализовать регистрацию и авторизацию.

Что сделать:

* регистрация пользователя;
* логин;
* JWT access token;
* refresh token;
* BCrypt для паролей;
* роли `USER` и `ADMIN`;
* миграции БД через Liquibase/Flyway;
* Swagger.

Результат:

```text
Пользователь может зарегистрироваться и войти в систему.
```

---

### Этап 3. User Service

**Цель:** реализовать профиль пользователя.

Что сделать:

* получить профиль;
* обновить профиль;
* хранить данные пользователя в PostgreSQL;
* добавить Redis cache;
* добавить проверку JWT;
* добавить DTO и MapStruct.

Результат:

```text
Есть отдельный сервис пользователей с кэшированием.
```

---

### Этап 4. Order Service

**Цель:** реализовать работу с заказами.

Что сделать:

* создать заказ;
* получить заказ по ID;
* получить список заказов пользователя;
* отменить заказ;
* добавить статусы заказа;
* отправлять событие `order.created` в Kafka.

Результат:

```text
Пользователь может создать заказ, а система отправляет событие в Kafka.
```

---

### Этап 5. Payment Service

**Цель:** реализовать оплату заказа.

Что сделать:

* получать событие `order.created`;
* создавать платёж;
* эмулировать успешную/неуспешную оплату;
* отправлять `payment.success` или `payment.failed`;
* добавить idempotency key;
* добавить outbox table.

Результат:

```text
Платёж обрабатывается асинхронно через Kafka.
```

---

### Этап 6. Связь Order + Payment

**Цель:** завершить бизнес-сценарий заказа.

Что сделать:

* `order-service` слушает события оплаты;
* если платёж успешен — статус заказа `PAID`;
* если ошибка — статус `FAILED`;
* добавить отмену заказа;
* добавить обработку повторных сообщений Kafka.

Результат:

```text
Заказ проходит полный путь: CREATED → PAID / FAILED.
```

---

### Этап 7. Notification Service

**Цель:** реализовать уведомления.

Что сделать:

* слушать события `payment.success`, `payment.failed`;
* отправлять mock email;
* сохранять историю уведомлений;
* добавить retry;
* добавить DLQ для неудачных сообщений.

Результат:

```text
Пользователь получает уведомление после оплаты.
```

---

### Этап 8. Analytics Service + ClickHouse

**Цель:** реализовать аналитику.

Что сделать:

* слушать события заказов и платежей;
* сохранять события в ClickHouse;
* создать таблицы `order_events`, `payment_events`;
* реализовать запросы:

  * дневная выручка;
  * средний чек;
  * количество заказов;
  * конверсия оплат;
  * топ пользователей.

Результат:

```text
Система собирает аналитику в ClickHouse.
```

---

### Этап 9. API Gateway

**Цель:** сделать единую точку входа.

Что сделать:

* настроить маршруты ко всем сервисам;
* добавить JWT-фильтр;
* добавить rate limiting через Redis;
* настроить CORS;
* вынести публичные и приватные endpoints.

Результат:

```text
Все запросы идут через API Gateway.
```

---

### Этап 10. Тестирование

**Цель:** покрыть проект тестами.

Что сделать:

* unit-тесты сервисного слоя;
* integration-тесты контроллеров;
* Testcontainers для PostgreSQL;
* Testcontainers для Kafka;
* Testcontainers для Redis;
* Testcontainers для ClickHouse;
* проверить основной сценарий заказа.

Результат:

```text
Проект имеет нормальное тестовое покрытие и проверяет бизнес-процессы.
```

---

### Этап 11. Мониторинг и логирование

**Цель:** сделать проект похожим на production.

Что сделать:

* добавить Spring Boot Actuator;
* подключить Prometheus;
* настроить Grafana;
* добавить structured logging;
* добавить traceId;
* подключить Loki или ELK;
* добавить OpenTelemetry + Jaeger.

Результат:

```text
Можно смотреть метрики, логи и трейсинг запросов.
```

---

### Этап 12. CI/CD

**Цель:** автоматизировать сборку.

Что сделать:

* GitHub Actions;
* запуск тестов при push;
* сборка Docker images;
* публикация images;
* проверка качества кода;
* отдельный pipeline для deploy.

Результат:

```text
Проект автоматически собирается и тестируется.
```

---

### Этап 13. Kubernetes

**Цель:** подготовить деплой.

Что сделать:

* `Deployment`;
* `Service`;
* `ConfigMap`;
* `Secret`;
* `Ingress`;
* health checks;
* readiness/liveness probes;
* ресурсы CPU/RAM;
* Helm Chart.

Результат:

```text
Проект можно развернуть в Kubernetes.
```

---

### Этап 14. Документация

**Цель:** подготовить проект для GitHub и резюме.

Что сделать:

* README;
* схема архитектуры;
* описание сервисов;
* описание Kafka events;
* инструкция запуска;
* примеры запросов;
* скриншоты Swagger/Grafana;
* блок “Что изучено”.

Результат:

```text
Проект выглядит как полноценный backend-проект для портфолио.
```

```text
1. Auth
2. Order
3. Payment
4. Kafka
5. Analytics + ClickHouse
6. Gateway
7. Tests
8. Docker
9. Monitoring
10. Kubernetes
```

Минимальный MVP:

```text
Auth → Order → Payment → Kafka → ClickHouse → Docker Compose
```
